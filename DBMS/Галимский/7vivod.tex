Мною была разработана база данных расписаний занятий студентов университета-. В ходе разработки была исследована предметная область. База была разработана по аналогии с расписанием занятий в СПбГПУ. На этом этапе выяснилось, что полностью рабочую самогенерирующуюся базу (автоматическое распределение занятий в наиболее удачном порядке) сделать невозможно. Для этого требуется учесть множество нюансов: занятия должны проходить в удобном порядке, как для преподавателя, так и для студентов. Так же занятия не должны быть разбросаны по всему кампусу. Поэтому решено было сделать базу данных - редактор для проверки на возможные пересечения занятий и прочих ошибок диспетчера. Основная задача данной базы: предоставление информации о расписании в наиболее удобном виде и простота программирования приложений, которые будут использовать данную базу.

Для удобства программирования была создана схема (см. п. \ref{sect:2.3}, \ref{sect:2.4}) и написана таблица параметров атрибутов базы данных (см. п. \ref{sect:3}). С помощью этих материалов была запрограммирована база данных (см. п. \ref{sect:4}). Для воссоздания реальной ситуации работы с базой данных был написан генератор случайных значений атрибутов таблиц (см. п.\ref{sect:5.2}). После этого для тестирования работоспособности базы были написаны запросы (см. п.  \ref{sect:6}).

Для решения поставленной задачи были использованы следующие материалы: Введение в системы баз данных (К.Дж. Дейт), лекционный материал, уроки по MySQL, справочник по MySQL

Используемое ПО: Microsoft Office Word 2007, Microsoft Visio 2010, Microsoft Visual Studio 2010, MySQL Server 5.5.

При программировании БД основной являлась проблема кодировок. Решена она была следующим способом: изменены кодировки самой базы данных на Windows ср1251 (при программировании БД в MySQL вначале введена строка <<create database 'dtbname1 default character set 'cpi25i';>>). Также изменены кодировки клиента, связи и вывода результатов на Windows cp866 (перед вводом запросов в консоль введено «set names ср866»). Неясен остался вопрос: почему при изменении кодировки клиента, связи и вывода результатов на такую же, как у базы данных, результат запросов выводится с ошибочной кодировкой.
